# Atrium Cockpit — Real Linear Data + Richer Surfaces

**Date:** 2026-06-08
**Status:** Approved (brainstorming) — ready for implementation plan
**Branch context:** `claude/vs-plugin-architecture-FyCiv` (VS Code extension POC under `extension/`)

## Problem

The cockpit webview renders `STUB_WAVES` — hardcoded fake wave/ticket data
(`extension/src/cockpit-html.ts`). The extension already reads real workspace
folders + branch from VS Code, but the SDLC content is invented. We want the
cockpit to show the **real Atrium Linear board** and to visualise sprint and
overall progress in a way that's true to how this project actually works
(waves, spikes that gate waves, UAT loop-backs).

## Key constraint (why "wire up the MCP" isn't literal)

The Linear MCP tools (`mcp__linear__*`) live in the **Claude Code session**, not
in the **VS Code extension host**. When the extension runs (F5 or installed) it
is a separate Node process with no access to those tools. So real Linear data
needs an explicit data path. Decision: **snapshot now, live source later**.

## Approach (decided)

- **Data source:** a committed JSON snapshot of the real board, generated by
  Claude via the Linear MCP. Real data today, no API key, point-in-time.
- **Live-ready shape:** the loader sits behind a `BoardSource` interface so a
  future `LinearSdkSource` (live `@linear/sdk` + API key) swaps in without
  touching the UI.
- **Click-through scope:** open-in-Linear + reveal-branch only. Triggering
  `/sprint` from a row is NOT feasible (a webview can't invoke a Claude Code
  skill) — that's Wave 1 CLI-bridge work, explicitly out of scope.

## Architecture

```
Linear MCP (Claude session, one-off)
        │  generate / refresh
        ▼
extension/src/atrium-board.json        ← committed snapshot (data, not code)
        │  import (resolveJsonModule)
        ▼
extension/src/board.ts                 ← BoardSource: load + validate + rollups
        │  buildInitPayload()
        ▼
extension/src/extension.ts (host)      ← postMessage("init", payload)
        │                                  + handles openLinear / revealBranch
        ▼
extension/webview/Cockpit.tsx (React)  ← renders rollup, active work, waves,
                                           segmented bars, spike-gate map
```

### Components

**`extension/src/atrium-board.json`** — the snapshot. Schema:

```jsonc
{
  "project": "Atrium",
  "generatedAt": "2026-06-08",          // stamped at generation time
  "spikes": [                            // gating spikes, surfaced on the map
    { "id": "STO-2146", "code": "T-110", "gatesWave": "Wave 1", "state": "todo" }
  ],
  "waves": [
    {
      "name": "Wave 0 · Visual layer",
      "label": "ATR Wave 0",
      "stage": "release",                // furthest stage this wave reached
      "passN": 1,                         // UAT loop-back count (1 = no bounce)
      "gatedBy": null,                    // spike code if blocked, else null
      "tickets": [
        {
          "id": "STO-2095",
          "title": "AppShell — folder canvas + rails",
          "url": "https://linear.app/x/issue/STO-2095",
          "priority": "high",
          "state": "done",               // todo | doing | done
          "spec": ["...acceptance criteria from description..."],
          "tests": { "passed": 0, "failed": 0, "missing": 0, "discovered": false },
          "activity": [
            { "kind": "pickup", "text": "...", "when": "May 4" }
          ]
        }
      ]
    }
  ]
}
```

**`extension/src/board.ts`** — pure, no `vscode` import (mirrors the existing
`cockpit-html.ts` split for testability):
- `interface BoardSource { load(): Board }` — `SnapshotSource` implements it by
  importing the JSON; `LinearSdkSource` is the future live impl (stub/not built).
- `computeRollup(board)` → `{ total, done, doing, todo, spikeCount, gatedWaves }`.
- `computeWaveProgress(wave)` → `{ done, doing, todo, total }` for the segmented bar.
- `resolveActiveTicket(board, branch)` → match branch name against ticket ids
  (e.g. `…sto-2181…` → STO-2181), fall back to first `doing` ticket, else null.

**`extension/src/extension.ts`** (host):
- `buildInitPayload()` reads from `board.ts` instead of `STUB_WAVES`; adds
  `rollup`, `activeTicket`, and `spikes` to the payload.
- New message handlers: `openLinear` (`url` → `vscode.env.openExternal`),
  `revealBranch` (best-effort, optional this slice).

**`extension/webview/Cockpit.tsx`** + new child components:
- `RollupBar` — project glance: counts by state + **segmented state bar**
  (done/doing/todo as colored segments) + spike count.
- `ActiveWork` — collapsible "what am I doing now" strip: active ticket, its
  branch, state, next-action hint.
- `SpikeGateMap` — wave sequence with ⚠ gate chips on blocked waves
  ("gated by T-110"). Can render inside the rollup region.
- `WaveSection` — gains a **segmented progress bar** (replaces bare %) and a
  **Pass-N / UAT-loop-back badge** when `passN > 1`.
- `TicketRow` — gains an `↗` open-in-Linear button (posts `openLinear`).
- `TicketDetail` tabs — filled with real `spec` / `activity`; Tests tab shows
  Linear-recorded counts or "not discovered yet" when `discovered: false`.

### Types (`extension/webview/types.ts`)

Extend: `Ticket` gains `url` + `tests.discovered`. New `Spike`, `Rollup`,
`WaveProgress`. `Wave` gains `label`, `stage`, `passN`, `gatedBy`.
`InitPayload` gains `rollup`, `activeTicket`, `spikes`, `generatedAt`.

## Snapshot generation (documented Claude task)

The snapshot is regenerated by a Claude session, not at runtime. A README
section (`extension/README.md`) documents the exact MCP queries + the JSON
schema so any future session can refresh it:
- `mcp__linear__list_issues` filtered per `ATR Wave *` label (paged/filtered to
  avoid the ~80k-char full dump — see CLAUDE.md gotcha).
- For each issue: id, title, url, priority, state, description→spec, and
  `mcp__linear__list_comments`→activity (pickup/plan/phase/close/commit).
- Stamp `generatedAt`. Write to `extension/src/atrium-board.json`.

## Visualisation decisions

**In scope (core):**
- Segmented state bar (done/doing/todo) — project-level in RollupBar and
  per-wave in WaveSection. Replaces the bare % bar.
- Spike-gate map — waves as a sequence with gate chips showing blockers.
- Per-wave Pass-N / UAT-loop-back badge.

**Fast-follows (NOT this spec):**
- Focused sprint board (one wave zoomed, priority queue, next-up highlighted).
- Throughput sparkline (closes/week from Linear `completedAt`).
- True test discovery (scan workspace for test files — that's STO-2186).

## Error handling

- Malformed/missing JSON → `board.ts` validates and the host posts an `init`
  with an empty board + an `error` field; webview shows a non-fatal "couldn't
  load board snapshot" banner rather than a blank screen.
- `resolveActiveTicket` returns null when nothing matches → Active Work strip
  hides cleanly.
- `openExternal` failure is swallowed host-side (logged, no crash).

## Testing

- `board.test.ts` — `computeRollup`, `computeWaveProgress`, `resolveActiveTicket`
  (branch-id match + doing fallback + null), JSON validation (good + malformed).
- `cockpit.test.tsx` — RollupBar renders segmented segments in correct ratios;
  SpikeGateMap shows gate chip on a blocked wave; Pass-N badge appears only when
  `passN > 1`; TicketRow open-in-Linear posts the right message; Tests tab shows
  "not discovered" when `discovered: false`.
- Existing `cockpit-html.test.ts` host CSP/HTML tests stay green.

## Vertical slices (build order)

1. **Snapshot + loader** — generate `atrium-board.json` (real data), build
   `board.ts` behind `BoardSource`, swap `buildInitPayload` off `STUB_WAVES`.
   Cockpit renders real waves/tickets. (Foundation — proves the data path.)
2. **Rollup bar + segmented bars** — RollupBar + per-wave segmented progress.
3. **Active Work strip** — branch→ticket resolution + the strip.
4. **Spike-gate map + Pass-N badge** — the critical-path visualization.
5. **Click-through + detail depth** — open-in-Linear button + real spec/activity
   tabs + Tests "not discovered" state.

Each slice ends green (tsc + vitest) and visually verifiable in F5.

## Out of scope (v0.x)

Live SDK fetching (interface only), `/sprint` trigger from rows, true test
discovery, sparkline/time-series, drag-to-reorder writes to Linear.
