# Atrium Cockpit — live mock-up extension (POC)

Proves the **VS Code extension** path for Atrium: keep the full editor, git,
terminal, extension ecosystem (incl. the Claude Code extension) and multi-root
folders that VS Code already ships, and add the Atrium **Linear / PRD / sprint
cockpit** on top as a React webview.

It runs against a **real snapshot of the Atrium Linear board** (`src/atrium-board.json`,
generated by a Claude session via the Linear MCP — see *Refreshing the board
snapshot* below) — **or**, if you set a Linear API key, it pulls the board
**live** from Linear behind the same `BoardSource` interface (`src/linear-source.ts`,
see [SETUP.md](SETUP.md)).

## What it shows

- An **Atrium** icon in the Activity Bar → a **Cockpit** view (sprint pipeline,
  wave cards with progress, ticket rows) populated from the real board.
- The same cockpit as a full editor tab: Command Palette → **Atrium: Open Dashboard**.
- The header shows your **real** open workspace folders (multi-root proof).
- Tickets inline-expand to **Spec / Tests / Activity** tabs (real acceptance
  criteria + comment timeline; Tests shows "not discovered yet" until test
  discovery lands).

## Run it (the F5 loop)

```bash
cd extension
bun install
```

Then **open the `extension/` folder in VS Code** (File → Open Folder…) and press
**F5**. A second VS Code window — the *Extension Development Host* — launches with
Atrium loaded. Click the Atrium icon in its Activity Bar.

The `build` task (Vite for the webview + `tsc` for the host) runs automatically
before launch. To iterate on the cockpit UI without restarting, run
`bun run watch:webview` in a terminal and use **Developer: Reload Window** in the
host.

## Layout

```text
extension/
  src/extension.ts        host: registers the view + command, loads the board, brokers messages
  src/board.ts            pure board model + BoardSource (SnapshotSource today, LinearSdkSource stub)
  src/atrium-board.json   committed snapshot of the real Atrium Linear board (copied to dist/ at build)
  src/cockpit-html.ts     CSP webview HTML + Tier-1 pipeline stages
  webview/                React cockpit (built by Vite → dist/webview)
    App.tsx               owns the host message channel
    Cockpit.tsx           the cockpit UI (pipeline, waves, ticket rows)
  media/atrium.svg        Activity Bar icon
```

## Refreshing the board snapshot

`src/atrium-board.json` is point-in-time. To refresh it, ask a Claude session
(with the Linear MCP authenticated) to regenerate it:

1. Page `mcp__linear__list_issues` with `project: "Atrium"`, `limit: 50`,
   following the `cursor` to exhaustion. Exclude `Canceled` / `Duplicate`.
2. For `Done` / `In Progress` / `In Review` issues only, pull
   `mcp__linear__list_comments` to build the `activity[]` timeline.
3. Map each issue to the schema in `src/board.ts` (`Board` → `waves[]` →
   `tickets[]`): `priority` 1/2/3/4 → `urgent/high/med/low`; state
   `Backlog|Todo → todo`, `In Progress → doing`, `In Review → review`,
   `Done → done`;
   `spec[]` = up to 3 acceptance-criteria bullets from the description;
   `tests` = `{0,0,0,discovered:false}` (real counts await test discovery).
4. Group by `ATR Wave *` label into the ordered `waves[]`; set `gatedBy` to the
   spike code for Waves 1/2/3 (`T-110`/`T-209`/`T-301`); `passN: 1`.
5. Write the JSON to `src/atrium-board.json` and run `bun run test` —
   `board.test.ts` + the live-file load will flag any schema drift.

The build copies it to `dist/` and the host reads it from there at runtime; a
malformed/missing file degrades to an empty board + a non-fatal banner, never a
blank screen.
